## Технические заметки

**Решение:** Зависимости в Middleware внедрять через `Invoke`, а не конструктор

**Почему:** Конструктор Middleware вызывается один раз — это singleton. Сервисы через `Invoke` создаются заново на каждый запрос — гарантирует свежий экземпляр

**Что я буду делать иначе:** Сразу смотреть как устроены другие Middleware в проекте перед написанием своего

## Что я выучил:

- **Marker interface** — пустой интерфейс как метка для автоматической регистрации через рефлексию
- **`IOptions<T>`** — стандартный способ внедрить конфиг в сервис через DI
- **Слои конфигов** — `appsettings.json` база, `appsettings.{env}.json` перезаписывает
- **Жизненный цикл Middleware** — конструктор singleton, `Invoke` на каждый запрос
- **Зона ответственности Middleware** — каждый отвечает за своё, не должен останавливать цепочку если это не его задача
- **`JsonDocument`** — читать конкретное поле из JSON без привязки к классу
